# CIFAR-100 分类项目——工程实现思路详解

本文档从"如果要从零搭建这个项目，每一步怎么想、为什么这么做"的角度，完整讲解整个工程的设计思路。

---

## 一、需求分析：我们要做什么

课程作业的核心要求：

1. 用 **MLP** 和 **CNN** 两种模型，分别在 CIFAR-100 数据集上做图像分类
2. 对比不同**优化器**（SGD、Adam）和不同**学习率**（0.1、0.01、0.001）的效果
3. 画出训练曲线（loss、accuracy），展示测试集预测结果
4. 输出一份汇总报告

算一下：2 个模型 × 2 个优化器 × 3 个学习率 = **12 组实验**。手动跑 12 次太麻烦，所以工程设计的第一个目标就是：**一键跑完所有组合，自动保存结果**。

## 二、整体架构：文件怎么组织

```text
深度学习/
├── pyproject.toml          ← 依赖管理（项目需要哪些库、什么版本）
├── train.py                ← 入口脚本：跑实验的"总指挥"
├── summarize_results.py    ← 报告脚本：汇总实验结果生成 Markdown
├── src/                    ← 核心代码包
│   ├── __init__.py         ← 包的入口，统一暴露函数
│   ├── models.py           ← 模型定义（MLP、CNN）
│   └── trainer.py          ← 训练逻辑、数据加载、可视化、工具函数
├── data/                   ← 数据集（自动下载，不提交到 git）
├── outputs/                ← 实验输出（每组实验一个子目录）
└── reports/                ← 报告和文档
```

### 为什么这么分

| 设计决策 | 理由 |
|---------|------|
| `src/` 做成一个包 | 所有核心逻辑集中管理，`train.py` 只做"调度"，不写具体实现 |
| 模型和训练逻辑分文件 | `models.py` 只管"网络长什么样"，`trainer.py` 管"怎么训练"，职责清晰 |
| `__init__.py` 统一导出 | `train.py` 只需 `from src import build_model, fit_model, ...`，不用关心内部文件结构 |
| `data/` 和 `outputs/` 写进 `.gitignore` | 数据和模型文件太大，不该提交到 git |
| `pyproject.toml` 管理依赖 | 用 `uv sync` 一键安装所有库，别人拿到代码能直接跑 |

## 三、数据流：一张图片从进入到出结果，经历了什么

```text
原始图片 (PIL Image, 32×32, 0-255)
    │
    ▼ transforms.RandomCrop + RandomHorizontalFlip  ← 只在训练时做，数据增强
    │
    ▼ transforms.ToTensor()                         ← 变成 [3, 32, 32] 的张量，值域 [0,1]
    │
    ▼ transforms.Normalize(mean, std)               ← 标准化到均值≈0，让梯度更稳定
    │
    ▼ DataLoader(batch_size=128, shuffle=True)      ← 128 张图打成一批，打乱顺序
    │
    ▼ model(inputs)                                 ← 前向传播，输出 [128, 100] 的得分矩阵
    │
    ▼ CrossEntropyLoss(logits, targets)             ← 对比预测和真实标签，算出一个数字（损失）
    │
    ▼ loss.backward()                               ← 反向传播，算出每个参数的梯度
    │
    ▼ optimizer.step()                              ← 根据梯度更新参数
    │
    ▼ 下一批 128 张图 → 重复以上过程
```

### 关键设计点

**1. 训练集 vs 验证集 vs 测试集为什么要分开？**

```text
50000 张训练图
    ├── 45000 张 → 训练集（用来训练模型）
    └── 5000 张  → 验证集（用来选"哪一轮的模型最好"）

10000 张测试图 → 测试集（最终评分，训练过程中完全不碰）
```

验证集的作用是**防止选错模型**。训练 10 轮，不一定第 10 轮最好——可能第 7 轮在验证集上效果最好（之后开始过拟合了）。所以每轮都在验证集上评估，保存最好的那一轮参数。

**2. 为什么训练集要数据增强，验证/测试集不做？**

训练时做 RandomCrop 和 RandomHorizontalFlip，相当于每次看到同一张图的"变体"——平移几个像素、左右翻转。这等价于人为增加数据量，减轻过拟合。但评估时必须用原始图片，否则结果不稳定。

**3. 为什么标准化？**

神经网络对输入数据的尺度敏感。像素值 [0, 255] 太大了，梯度会爆炸。标准化到均值≈0、标准差≈1 后，梯度更稳定，训练更快。

## 四、模型设计思路（models.py）

### 为什么把模型单独放一个文件

模型是"能独立存在"的东西——它只定义网络结构，不依赖训练逻辑。今后如果要换一个新模型（比如 ResNet），只需要在 `models.py` 里加一个类，再在 `build_model()` 里注册一行就行，不用动任何训练代码。

### 工厂模式：`build_model()`

```python
def build_model(model_name: str, num_classes: int = 100) -> nn.Module:
    model_name = model_name.lower()
    if model_name == "mlp":
        return MLPClassifier(num_classes=num_classes)
    if model_name == "cnn":
        return SimpleCNN(num_classes=num_classes)
    raise ValueError(f"Unsupported model: {model_name}")
```

为什么不直接在 `train.py` 里写 `if model_name == "mlp": model = MLPClassifier()`？因为**把"选哪个模型"这个决策封装成一个函数**，外面只传名字就行，不需要知道具体的类名。好处：

- 新增模型只改一个文件
- `train.py` 里的循环代码完全通用，不需要为每种模型写不同的分支

### MLP 的设计

```text
输入 [batch, 3, 32, 32]
  │
  ▼ Flatten → [batch, 3072]
  ▼ Linear(3072, 1024) → ReLU → Dropout(0.3)
  ▼ Linear(1024, 512)  → ReLU → Dropout(0.3)
  ▼ Linear(512, 100)
  │
输出 [batch, 100]
```

三层全连接，逐步压缩维度。用 `nn.Sequential` 串起来——一个最简洁的写法，不需要在 `forward()` 里手动一层一层调用。

### CNN 的设计

```text
输入 [batch, 3, 32, 32]
  │
  ▼ 第一阶段：Conv(3→64) → BN → ReLU → Conv(64→64) → ReLU → MaxPool → Dropout2d
  │  → [batch, 64, 16, 16]
  │
  ▼ 第二阶段：Conv(64→128) → BN → ReLU → Conv(128→128) → ReLU → MaxPool → Dropout2d
  │  → [batch, 128, 8, 8]
  │
  ▼ 第三阶段：Conv(128→256) → BN → ReLU → MaxPool
  │  → [batch, 256, 4, 4]
  │
  ▼ Flatten → Linear(4096, 512) → ReLU → Dropout → Linear(512, 100)
  │
输出 [batch, 100]
```

设计思路是经典的 VGG 风格：

| 设计选择 | 原因 |
|---------|------|
| `padding=1` + 3×3 卷积 | 保持卷积前后尺寸不变，只让 MaxPool 负责缩小 |
| 通道数 64→128→256 | 浅层检测简单特征（种类少），深层检测复杂特征（种类多），需要更多通道 |
| 每个阶段两层卷积 | 单层卷积感受野太小（3×3），两层叠加相当于 5×5，能看到更大范围 |
| BatchNorm 放在每个阶段的第一个卷积后 | 稳定训练，加速收敛 |
| Dropout2d 放在 MaxPool 后 | 在特征图缩小后再丢弃，效果更好 |
| 分成 `self.features` 和 `self.classifier` | 特征提取和分类决策分开，清晰 |

### 为什么 CNN 分成 `features` 和 `classifier` 两个 Sequential

这不仅是代码清晰的问题。实际工程中经常需要**单独取出特征**——比如做迁移学习时，冻结 `features`、只训练 `classifier`。提前把结构分好，将来改起来很方便。

## 五、训练逻辑设计（trainer.py）

### 这个文件为什么这么长

它承担了除模型定义外的**所有职责**：

```text
trainer.py
├── 环境配置（matplotlib 缓存路径）
├── 工具函数
│   ├── set_seed()        → 固定随机种子
│   ├── get_device()      → 自动选 GPU/CPU
│   └── _subset_dataset() → 截取数据子集
├── 数据加载
│   └── get_cifar100_loaders() → 下载、预处理、切分、创建 DataLoader
├── 优化器
│   └── build_optimizer() → 根据名字创建 SGD/Adam
├── 训练核心
│   ├── _run_epoch()      → 跑一个 epoch（训练或评估）
│   └── fit_model()       → 完整训练流程（多 epoch + 最优模型选择）
├── 可视化
│   ├── plot_curves()     → 画 loss/accuracy 曲线
│   ├── plot_predictions()→ 画预测结果网格图
│   └── _denormalize()    → 把标准化后的图片还原回可显示的范围
└── IO
    └── save_json()       → 保存指标到 JSON
```

### `_run_epoch()` 的巧妙设计

这个函数**同时处理训练和评估**，靠一个参数区分：

```python
def _run_epoch(model, loader, criterion, device, optimizer=None):
    is_train = optimizer is not None  # 传了 optimizer 就是训练，没传就是评估
    model.train(is_train)             # 控制 Dropout 和 BatchNorm 的行为
    
    with torch.set_grad_enabled(is_train):  # 评估时关闭梯度计算
        for inputs, targets in loader:
            if is_train:
                optimizer.zero_grad(set_to_none=True)  # 清梯度
            
            logits = model(inputs)
            loss = criterion(logits, targets)
            
            if is_train:
                loss.backward()    # 反向传播
                optimizer.step()   # 更新参数
```

**为什么不分成 `train_epoch()` 和 `eval_epoch()` 两个函数？** 因为两者的核心逻辑（前向传播 → 算损失 → 统计指标）完全一样，只是训练多了"清梯度 → 反向传播 → 更新"三步。合并成一个函数可以避免重复代码。

### `fit_model()` 的训练循环

```python
def fit_model(model, train_loader, val_loader, test_loader, optimizer, device, epochs):
    best_val_acc = -1.0
    best_state = {}
    
    for epoch in range(epochs):
        train_loss, train_acc = _run_epoch(... , optimizer)  # 训练
        val_loss, val_acc = _run_epoch(...)                  # 验证（不传 optimizer）
        
        if val_acc > best_val_acc:         # 这一轮比之前都好
            best_val_acc = val_acc
            best_state = 模型参数的深拷贝   # 保存下来
    
    model.load_state_dict(best_state)      # 恢复到最好的那一轮
    test_loss, test_acc = _run_epoch(...)   # 用最好的模型测一次
```

关键设计：

| 设计点 | 原因 |
|-------|------|
| 记录 history | 每个 epoch 的 loss 和 acc 都存下来，用于画曲线 |
| best_state 深拷贝 | `model.state_dict()` 返回的是引用，必须 `.clone()` 到 CPU，否则后续训练会覆盖 |
| 训练结束后加载 best_state | 最后一轮不一定最好，我们要的是验证集上最优的那一轮 |
| 在 best_state 上做测试集评估 | 确保报告的测试准确率对应的是最好的模型，而不是最后一轮 |

### `build_optimizer()` 的工厂模式

和 `build_model()` 一样的思路——用字符串名字创建优化器。`train.py` 的循环里只需要写：

```python
optimizer = build_optimizer(optimizer_name, model, lr)
```

不需要关心 SGD 和 Adam 的参数有什么不同。

## 六、实验调度设计（train.py）

这是整个项目的**总指挥**，它自己不实现任何逻辑，只做调度：

```text
train.py 的流程：

1. 解析命令行参数（--models, --optimizers, --learning-rates, --epochs 等）
2. 加载数据（调 get_cifar100_loaders）
3. 生成实验组合（笛卡尔积）
4. 循环跑每组实验：
   a. 创建模型（调 build_model）
   b. 创建优化器（调 build_optimizer）
   c. 训练+评估（调 fit_model）
   d. 保存模型、曲线图、预测图、JSON 指标
5. 汇总所有实验结果到 CSV
```

### 为什么用笛卡尔积

```python
experiments = [
    (model_name, optimizer_name, lr)
    for model_name in args.models
    for optimizer_name in args.optimizers
    for lr in args.learning_rates
]
```

这行代码自动生成所有组合。好处是**完全由命令行参数控制**——如果只想跑 CNN + Adam，改命令就行：

```bash
uv run train.py --models cnn --optimizers adam --learning-rates 0.001
```

不需要改任何代码。

### 每组实验独立一个目录

```text
outputs/
├── mlp_sgd_lr0.1/
│   ├── best_model.pt
│   ├── loss_curve.png
│   ├── prediction_examples.png
│   └── metrics.json
├── mlp_sgd_lr0.01/
│   └── ...
├── cnn_adam_lr0.001/
│   └── ...
└── summary.csv
```

好处：

- 每组实验的结果完全隔离，不会互相覆盖
- 文件夹名字就是实验配置，一目了然
- `metrics.json` 既存了最终指标，也存了每个 epoch 的详细记录（`history`），方便事后分析
- `summary.csv` 是所有实验的汇总，用 Excel 打开就能排序、对比

### 命令行参数的设计

所有超参数都暴露为命令行参数，每个都有**合理的默认值**：

```python
parser.add_argument("--epochs", type=int, default=10)
parser.add_argument("--batch-size", type=int, default=128)
parser.add_argument("--subset-ratio", type=float, default=1.0)
```

`--subset-ratio` 是一个实用技巧：设成 0.05 就只用 5% 的数据，几分钟跑完，用于**快速验证代码是否正确**。确认没 bug 后再设回 1.0 跑完整实验。

## 七、报告生成设计（summarize_results.py）

这是一个**完全独立的脚本**，和训练脚本解耦：

```text
train.py 产出 → outputs/各实验/metrics.json
                        ↓
summarize_results.py 读取所有 metrics.json
                        ↓
                  reports/自动实验报告.md
```

为什么不把报告生成放在 `train.py` 里？因为：

- 训练可能中途失败或分批跑，报告应该能在任何时候独立生成
- 可以反复生成报告（调格式、加分析），不需要重新训练
- 职责分离：`train.py` 管训练，`summarize_results.py` 管报告

## 八、工程细节和最佳实践

### 1. 随机种子

```python
set_seed(42)
```

固定所有随机源（Python、NumPy、PyTorch CPU/GPU），保证**同样的代码 + 同样的参数 = 同样的结果**。这对科学实验至关重要——如果结果不可复现，就无法对比。

### 2. 设备自动选择

```python
def get_device():
    if torch.backends.mps.is_available():   # Mac M 系列芯片
        return torch.device("mps")
    if torch.cuda.is_available():           # NVIDIA GPU
        return torch.device("cuda")
    return torch.device("cpu")              # 兜底
```

不硬编码设备，在任何机器上都能跑。

### 3. Matplotlib 缓存处理

```python
PROJECT_CACHE = PROJECT_ROOT / ".cache"
os.environ.setdefault("MPLCONFIGDIR", str(PROJECT_CACHE / "matplotlib"))
```

某些环境（比如学校服务器）用户主目录不可写，matplotlib 会报错。把缓存目录指向项目内的 `.cache/`，彻底避免权限问题。

### 4. `.gitignore` 的设计

```text
.venv/         ← 虚拟环境（几百 MB，每台机器不同）
.cache/        ← matplotlib 缓存
__pycache__/   ← Python 编译缓存
data/          ← CIFAR-100 数据集（~170 MB，不该提交）
outputs/       ← 模型文件和图片（太大，且可重新生成）
```

只提交**代码和文档**，不提交可重新生成的数据和结果。

### 5. `pyproject.toml` 的依赖管理

```toml
dependencies = [
    "matplotlib>=3.8.0",
    "torch>=2.2.0",
    ...
]
```

用最小版本约束（`>=`），不锁死具体版本。配合 `uv.lock` 文件实现精确复现——`uv sync` 会读 lock 文件安装完全一致的版本。

### 6. `__init__.py` 的统一导出

```python
from .models import build_model
from .trainer import (
    build_optimizer,
    fit_model,
    get_cifar100_loaders,
    ...
)
```

外部使用时只需 `from src import xxx`，不需要知道函数在 `models.py` 还是 `trainer.py` 里。如果未来重构了文件结构（比如把 `trainer.py` 拆成多个文件），只需改 `__init__.py`，`train.py` 不用动。

## 九、执行流程总览

把整个项目串起来看，从头到尾的执行流程是：

```text
用户执行: uv run train.py --epochs 10

① 解析参数
② set_seed(42) → 固定随机
③ get_device() → 选 GPU/CPU
④ get_cifar100_loaders() → 下载数据、预处理、切分、创建 DataLoader
⑤ 生成 12 组实验组合

对每组实验 (model, optimizer, lr):
  ⑥ build_model("cnn") → 创建一个全新的 CNN
  ⑦ build_optimizer("adam", model, lr=0.001) → 创建优化器
  ⑧ fit_model() → 训练 10 个 epoch
     每个 epoch:
       _run_epoch(train_loader, optimizer)  → 训练（更新参数）
       _run_epoch(val_loader)               → 验证（不更新）
       if val_acc > best → 保存参数快照
     训练结束 → 加载最佳参数 → 测试集评估
  ⑨ 保存: best_model.pt, loss_curve.png, prediction_examples.png, metrics.json

⑩ 汇总所有实验 → summary.csv
⑪ 打印排名表

---

用户执行: uv run summarize_results.py

① 读取 outputs/*/metrics.json
② 按 test_acc 排序
③ 生成 Markdown 报告 → reports/自动实验报告.md
```

## 十、如何扩展这个项目

这个工程结构的扩展性很好，常见的扩展方式：

| 扩展需求 | 改哪里 | 怎么改 |
|---------|--------|-------|
| 加一个新模型（如 ResNet） | `models.py` | 新增一个类 + 在 `build_model()` 里注册 |
| 加一个新优化器（如 AdamW） | `trainer.py` | 在 `build_optimizer()` 里加一个分支 |
| 加学习率调度器 | `trainer.py` | 在 `fit_model()` 的 epoch 循环里加 `scheduler.step()` |
| 换数据集（如 CIFAR-10） | `trainer.py` | 修改 `get_cifar100_loaders()`，换数据集和均值/标准差 |
| 加 TensorBoard 日志 | `trainer.py` | 在 `fit_model()` 里加 `SummaryWriter` |
| 支持断点续训 | `train.py` | 保存/加载 optimizer 和 epoch 状态 |

每种扩展只动一个文件，不会牵连其他部分——这就是良好架构的价值。
